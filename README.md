# Campus Secondhand Platform

校园二手闲置交易平台是一个面向高校学生的全栈交易系统，围绕“学生发布闲置商品、买家浏览下单、线下交割、平台审核与仲裁”的真实业务场景设计。项目覆盖学生移动端、管理后台和后端服务，包含商品、订单、支付、收藏、举报、评价、消息、公告、信用分和争议处理等模块。

本仓库作为熊银滢的个人项目展示，重点呈现完整业务系统的设计思路、后端业务建模、移动端接口对接、订单支付流程和前后端协同实现能力。

## 技术栈

| 模块 | 技术栈 | 说明 |
| ---- | ---- | ---- |
| `ruoyi-vue/` | Spring Boot 3.5、Spring Security、JWT、MyBatis、Redis、Quartz、Druid、Swagger | 后端服务，提供管理后台 API 与学生端 API |
| `ruoyi-vue3-frontend/` | Vue 3、Vite、Pinia、Element Plus、Axios、ECharts | 管理后台，面向平台运营与审核人员 |
| `flutter-app/` | Flutter、Dart、HTTP、SharedPreferences、WebView | 学生移动端，面向买家与卖家 |
| MySQL | 关系型数据建模 | 存储用户、商品、订单、评价、举报、争议等业务数据 |
| Redis | 缓存与登录态 | 缓存 Token、验证码、字典等高频数据 |

## 系统架构

项目采用前后端分离架构。后端基于 RuoYi Spring Boot 体系扩展业务模块，同时服务管理后台和学生移动端。

```text
Flutter 学生端
    |
    | /app/*
    v
Spring Boot 后端服务  ---- MySQL
    |                  ---- Redis
    |
    | /trade/* /ai/* /system/*
    v
Vue 3 管理后台
```

后端按 Controller、Service、Mapper、Domain 分层组织。Controller 负责参数接收和权限控制，Service 承载业务规则和状态流转，Mapper 通过 MyBatis 访问数据库，Domain / DTO / VO 负责数据结构隔离。

### 学生移动端

- 注册、登录、退出登录、Token 自动保存与会话恢复
- 首页商品推荐、分类浏览、关键词搜索、商品列表分页加载
- 商品详情查看、图片展示、卖家信息展示
- 商品发布、图片上传、我的发布、上架/下架状态展示
- 收藏商品、取消收藏、收藏列表
- 创建订单、买入订单、卖出订单、订单详情
- 支付宝沙盒支付页面、支付结果查询、支付回调后状态同步
- 退款申请、卖家处理退款、订单完成确认
- 商品举报、我的举报、争议提交、争议列表
- 评价发布、评价列表、公告列表与公告详情
- 消息会话和聊天页面

### 管理后台

- 管理员登录、动态路由、按钮级权限控制
- 商品分类管理、学生用户管理
- 闲置商品管理、商品审核、商品图片管理
- 订单管理、订单日志查看、支付状态跟踪
- 举报管理、争议仲裁管理、评价管理
- 公告管理、站内消息管理
- AI 商品审核记录、AI 估价记录、争议分析记录
- 信用分管理和信用流水查看
- 数据统计报表，辅助观察商品、订单和平台运营情况

### 后端服务

- 管理端与学生端使用不同认证入口，Token 逻辑隔离
- 基于 Spring Security + JWT 的无状态认证
- 基于 MyBatis 的业务表 CRUD、分页查询和复杂条件查询
- Swagger 接口文档，便于前后端联调
- Redis 缓存 Token、验证码和系统字典
- Quartz 定时任务处理支付超时取消、收货超时确认、信用限制解除等场景
- Druid 数据源监控
- 统一响应结构、统一异常处理、操作日志和权限注解

## 核心业务流程

### 商品发布与审核

```text
学生发布商品
  -> 商品进入待审核状态
  -> 管理员在后台审核
  -> 审核通过后上架展示
  -> 审核拒绝则返回拒绝状态
```

商品状态设计：

```text
0 待审核
1 已上架
2 审核拒绝
3 已下架
4 已售出
```

这样设计可以避免用户发布内容直接进入交易市场，同时也方便后台对违规商品进行拦截和追踪。

### 下单与支付

```text
买家选择商品
  -> 创建待支付订单
  -> 调起支付宝沙盒支付
  -> 支付成功回调后更新支付状态
  -> 订单进入待收货/待交割
  -> 买卖双方线下交割
  -> 买家确认完成或系统超时自动确认
```

订单状态和支付状态分开设计。订单状态描述业务进度，支付状态描述资金动作，这样可以更清楚地处理“订单已取消但支付中断”“已支付但待交割”“退款中”等复杂场景。

订单状态：

```text
0 待支付
2 待收货/待交割
3 已完成
4 已取消
5 争议中
6 退款中
7 已退款
```

支付状态：

```text
0 未支付
1 支付中
2 已支付
3 支付失败
4 已退款/已取消
```

### 退款与争议处理

```text
买家申请退款
  -> 卖家同意或拒绝
  -> 同意后调用退款逻辑并更新订单
  -> 拒绝后买家可提交争议
  -> AI 分析辅助仲裁
  -> 管理员最终处理
```

退款状态：

```text
0 无退款
1 买家申请待卖家处理
2 已同意退款中
3 已退款
4 卖家拒绝
```

争议状态：

```text
0 待 AI 分析
1 AI 分析中
2 等待人工仲裁
3 已处理
```

争议模块把自动分析和人工仲裁结合起来，核心目的不是完全替代人工，而是帮助管理员快速了解订单、举报理由、举证信息和建议处理方向。

## 数据设计原则

- 学生账号使用独立业务表，不复用后台 `sys_user`，避免管理端账号和学生端账号权限混杂。
- 业务表统一使用逻辑删除字段 `del_flag`，避免误删交易、订单、举报、评价等可追溯数据。
- 商品、订单、支付、退款、争议分别维护状态字段，降低单一状态枚举过度膨胀带来的复杂度。
- 订单表保存支付流水号、支付金额、支付时间、退款流水号等字段，方便后续对账与问题排查。
- 业务实体继承通用审计字段，保留创建人、创建时间、更新人、更新时间。
- 管理后台写操作通过操作日志记录，便于追踪审核、仲裁、调整信用分等敏感动作。

## 接口设计原则

- 管理端接口使用 `/trade/*`、`/ai/*` 等前缀，依赖后台登录态和权限码。
- 学生端接口使用 `/app/*` 前缀，移动端登录、注册、公开商品列表等接口可匿名访问，其余接口需要学生端 Token。
- Controller 层只做参数接收、权限校验和响应包装，核心业务规则沉到 Service 层。
- DTO 用于接收前端输入，VO 用于返回前端展示数据，避免直接暴露数据库实体。
- 接口通过 Swagger UI 提供文档，减少前后端口头约定。

## 安全与可靠性设计

- JWT 无状态认证，管理端和学生端 Token 隔离。
- 支付宝回调接口单独放行，并在业务层校验回调参数和订单状态。
- Redis 缓存验证码、Token 和字典，减少数据库压力。
- 定时任务兜底处理长期未支付订单和超时未确认订单，避免订单状态悬挂。
- 逻辑删除保护关键业务数据，便于后续审计和恢复。
- 后端配置通过环境变量注入，密钥、数据库密码和支付私钥不提交到仓库。

- 完整覆盖二手交易平台核心链路：发布、审核、浏览、下单、支付、交割、评价、举报、仲裁。
- 管理端和学生端账号体系隔离，权限边界清晰。

- 后端在 RuoYi 基础上扩展业务模块，复用权限、日志、分页、字典、Swagger 等成熟能力。

## 本地启动

### 后端服务

环境要求：JDK 17、Maven、MySQL 8、Redis。

```bash
cd ruoyi-vue
mvn -pl ruoyi-admin -am spring-boot:run
```

启动后可访问：

- Swagger UI: `http://localhost:8080/swagger-ui.html`
- Druid Monitor: `http://localhost:8080/druid/`

数据库脚本位于 `ruoyi-vue/sql/`，请按基础表、种子数据和增量脚本顺序导入。

### 管理后台

环境要求：Node.js 18+。

```bash
cd ruoyi-vue3-frontend
npm install
npm run dev
```

开发环境默认通过 `/dev-api` 代理到本地后端 `http://localhost:8080`。

### Flutter 学生端

环境要求：Flutter SDK 3+。

```bash
cd flutter-app
flutter pub get
flutter run
```
